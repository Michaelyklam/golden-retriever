This file captures the authoritative external sources we should treat as the recipe for the full-scale `golden-retriever` effort. It supersedes the small local proxy experiments as the planning baseline.

## Context-1 training recipe facts to copy

From the technical report and model card:

- Model role: retrieval subagent, not answer generator.
- Base model: `gpt-oss-20b`; Context-1 was trained with SFT + RLVR/CISPO using LoRA.
- Synthetic task domains: web, finance/SEC, legal/patents, email.
- Public data-gen repo domains:
  - `agentic_search_data_gen.domains.web`
  - `agentic_search_data_gen.domains.sec`
  - `agentic_search_data_gen.domains.patents`
  - `agentic_search_data_gen.domains.epstein`
- Data-gen pattern:
  1. gather supporting documents;
  2. generate obfuscated clues, question, and answer;
  3. verify quote/task validity;
  4. optionally mine and verify distractors;
  5. optionally recursively chain/extend tasks to control hop count.
- Agent tools:
  - `search_corpus(query)`: hybrid BM25 + dense vector search, RRF fusion, top-50 candidates, reranked into a token budget;
  - `grep_corpus(pattern)`: regex search, up to 5 matching chunks;
  - `read_document(doc_id)`: read full content/chunks for a chosen document;
  - `prune_chunks(chunk_ids)`: remove selected chunks from active model context while preserving full trajectory for reward computation.
- Harness behavior:
  - observe → infer → act state-machine loop;
  - dedupe previously encountered chunk IDs across search calls;
  - append token-usage visibility after each turn;
  - soft token threshold encourages pruning/concluding;
  - hard token cutoff rejects non-prune tool calls;
  - full unpruned trajectory remains available for reward computation.
- SFT warmup:
  - generated by rolling out stronger models through the full agent loop;
  - retains full trajectories, not just final answers;
  - filters based on trajectory recall and output recall;
  - keeps high-recall trajectories in full;
  - samples low-recall but well-formed trajectories at diminishing probability;
  - caps zero-recall traces around 5%;
  - excludes traces with high trajectory recall but low output recall to avoid teaching poor final selection;
  - discards malformed outputs.
- RLVR/CISPO:
  - on-policy LoRA training;
  - 128 queries per step × 8 rollouts = 1,024 trajectories/step in the report;
  - groups with identical rewards are discarded;
  - 5 epochs, about 300 possible steps, convergence around 230 steps;
  - reward combines recall-heavy F-score, trajectory recall, final-answer-found bonus, repeated-prune penalty, and turn-count penalty;
  - curriculum shifts from easier/lower-hop tasks to harder/multi-hop tasks and from recall-heavy toward more precision-aware scoring.

## Public benchmark suite definitions from the report

| Suite | Report description | Reproducibility implication |
|---|---|---|
| Generated web | BrowseComp-like generated web tasks, chained up to 4 hops. | We can generate compatible tasks from `context-1-data-gen` and hold out a split. |
| Generated finance / SEC | 2025 public SEC filings, chained up to 3 hops. | Requires EDGAR identity, embeddings, indexing, and generated held-out split. |
| Generated legal / patents | USPTO office-action rejection tasks where prior art is ground truth. | Requires USPTO/SearchAPI/Datalab-style extraction pipeline or a local equivalent. |
| Generated email / Epstein | Epstein email tasks over Epstein + Enron noisy corpus. | Data source is public/HF/GDrive per data-gen README; needs static corpus snapshot. |
| BrowseComp+ | Static-corpus version of BrowseComp-like deep research tasks. | Need dataset access or an equivalent static snapshot; live web BrowseComp is not comparable. |
| Seal-0 | 111 curated questions; evaluate whether positive URLs are identified. | Needs dataset/positive URL list and browsing/scraping adapter. |
| LongSeal | 254 needle-in-haystack questions with large retrieved document sets; chunk into 512-token chunks. | Good first full public adapter because it can be a fixed-corpus retrieval benchmark. |
| FRAMES | Wikipedia-derived multi-hop benchmark; positive Wikipedia URLs scored for recall. | Need Serper/Wikipedia API adapter and URL normalization/filtering. |
| HotpotQA | Simpler multi-hop retrieval benchmark; models mostly saturate it. | Useful sanity benchmark, not enough for parity. |
| HLE search condition | Search subagent retrieves docs for Opus-style answerer; compare answer accuracy lift vs no-search baseline. | Optional later-stage benchmark; requires answerer model and source access. |

## Current blockers / unknowns

- Chroma's Hugging Face card states the full agent harness and evaluation code are **not yet public**. We must implement a compatible harness ourselves and label deviations.
- Firecrawl/web_search is not configured in this Hermes environment, but direct terminal HTTP fetches worked for the technical report, GitHub repo, and Hugging Face model card.
- Full data-gen requires external services/API keys: Anthropic, OpenAI embeddings, ChromaDB, Serper/Jina, Baseten reranker, USPTO/SearchAPI/Datalab for patents. We should use local/static substitutes where possible, but record every deviation.
- `golden-retriever` currently has small local proxy datasets and no full observe/reason/act multi-turn training harness. That is the next major implementation slice.
